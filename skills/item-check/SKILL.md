# item-check — 이 아이템, 검증할 가치가 있나

## 이 스킬이 존재하는 이유 (실제 시행착오)
> 한 팀이 아이템의 수요를 "있는 것"으로 가정한 채 랜딩페이지를 12개 만들었다.
> 그 "수요 사례"는 나중에 **가상 예시**로 판명됐다. 2개월이 날아갔다.
> → 만들기 전에, **아이템 자체가 검증할 가치·검증 가능성이 있는지**부터 냉정하게 본다.

이 스킬은 아이템을 **통과시키는 게 목적이 아니라, 착수 전에 약한 곳을 드러내는 게 목적**이다.
칭찬하지 말고, 반박하라.

## 판정 기준 6개 (각 항목에 근거 태그: ✅검증 / ◐추정 / ❌가정)
1. **문제 실재성** — 진짜 아픈 문제인가, "있으면 좋은"(vitamin) 수준인가? 누가 이걸로 밤을 새우나?
2. **수요 증거** — 수요의 근거가 실제인가, "있을 것"이라는 가정인가?
   ⚠️ 여기서 가장 많이 속는다. 인용하는 사례가 **실제 인입/계약/인터뷰**인지, 머릿속 예시인지 반드시 구분.
3. **차별성/엣지** — 기존 대안 대비 이길 이유가 있나, me-too인가? "우리도 할 수 있다"는 엣지가 아니다.
   → 경쟁자가 이미 같은 걸 더 크게 하고 있지 않은지 **반박**해본다.
4. **타겟 도달성** — 살 사람에게 실제로 닿을 채널이 있나?
5. **지불의사 개연성** — 돈(또는 시간·평판)을 낼 만한 대상인가? "무료면 쓰겠다"는 신호가 아니다.
6. **리스키스트 가정 + 검증 가능성** — 이게 틀리면 죽는 가정 1개를 뽑고, 그걸 **싸게** 테스트할 수 있나?
   랜딩/fake-door로 되는가, 아니면 다른 방법(선판매·인터뷰)이 필요한가?

## 순서
1. 아이템을 한두 문장으로 받는다. 모호하면 되묻는다(대상·문제·과금).
2. 기준 6개를 하나씩 점검하고 **근거 태그**를 붙인다. 근거를 못 대면 자동 ❌가정.
3. **리스키스트 가정 1개**를 명시한다.
4. **판정**을 내린다:
   - ✅ **적절** — 지금 검증 착수 OK
   - ◐ **조건부** — 아래 항목 보완 후 착수
   - ❌ **부적절** — 지금 랜딩 만들지 마라. 먼저 이걸 해결/재정의하라
5. `item-check.md`로 저장.
