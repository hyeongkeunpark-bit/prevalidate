# key-copy — 캔버스 내용 × 레퍼런스 보이스 = 키카피 N개

키카피 = 랜딩의 **핵심 한 줄(Hero 헤드라인)**. 두 축을 곱한다:
- **What(무엇을 말할까)** = 린캔버스의 **UVP·Problem·Customer 칸**
- **How(어떻게 말할까)** = **레퍼런스 사이트별 보이스** ← 다양성의 주력 축

> 왜 레퍼런스 보이스인가: 실제 사이트 4개를 주고 "우리 기획에 맞춰 새로 써줘" 했을 때
> **스타일이 다양한 키카피**가 나와서 가장 쓸모 있었다. 그 경험을 그대로 스킬화한 것.
> (이 프로젝트의 "제약(레퍼런스)으로 준다" 방법론과 동일)

## 전제
- `lean-canvas.md`를 먼저 읽는다. 없으면 `/item-check` 먼저.
- 판정이 ❌(아직 이르다)면 **카피를 뽑지 않는다.** 캔버스 빈칸부터.
- **캔버스가 갱신되면 다시 돌린다.** 특히 UVP·Problem·Customer가 바뀌면 카피가 확 달라짐(Cost·Metrics 변경은 거의 무영향 — 정상).

## N & 보이스 정하기
- 사용자가 개수를 주면 그대로. 안 주면 **기본 5개.**
- **모드 A (주력) — 레퍼런스 보이스**: 사용자가 사이트/브랜드 N개를 준다
  → 각 사이트 카피를 읽어 **보이스 추출**(문장 길이·구어체/단정체·영문 라벨·감정 vs 스펙 지향·구조)
  → 우리 캔버스 메시지를 **그 보이스로** 한 줄씩 = N개.
  - URL이면 읽어서(WebFetch/browse) 실제 헤드라인·톤을 뜬다. 못 읽으면 그 브랜드의 알려진 보이스로.
- **모드 B (폴백) — 레퍼런스 없을 때**: 어울리는 레퍼런스 브랜드 후보를 제안해 고르게 하거나,
  소구 앵글 팔레트(문제/성과/타이밍/정체성/가격/반전)로 변형.

## 순서
1. `lean-canvas.md`와 아이템을 읽는다(UVP=핵심 메시지, Problem/기존대안=고통·앵커, Customer=톤).
2. 레퍼런스 N개 확보(모드 A) 또는 후보 제안(모드 B).
3. 레퍼런스마다 보이스 추출 → 같은 캔버스 메시지를 그 보이스로 키카피 1개.
4. 각 카피에 **[레퍼런스 보이스] + [출처 칸] + 왜** 를 붙인다.
5. 검증 안 된 수치는 `[플레이스홀더]`.
6. `key-copy.md`로 저장.

## 산출물 템플릿 (`key-copy.md`)
```
## 아이템: (한 문장)  |  N = (개수)  |  캔버스 기준: (UVP 요약)

| # | 키카피 | 레퍼런스 보이스 | 출처 칸 / 왜 |
|---|--------|-----------------|--------------|
| 1 | "…"    | Linear식 단정·영문라벨 | UVP / …  |
| 2 | "…"    | Toss식 구어체·공감      | Problem / … |
| … |        |                 |              |

추천 우선 테스트 2~3개: (보이스가 가장 갈리는 것들)
```
